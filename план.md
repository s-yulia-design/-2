# Единый план разработки приложения в Cursor для анализа целесообразности подключения новых населённых пунктов интернет-провайдером

## 1. Идея приложения

Приложение должно работать как аналитический дашборд для интернет-провайдера.

Пользователь вводит название населённого пункта Волгоградской области, а система помогает собрать, структурировать и проанализировать данные для принятия решения:

**стоит ли подключать населённый пункт к услугам интернет-провайдера, в какой очередности это делать и при каких условиях проект станет выгодным.**

Система должна:

1. хранить карточки населённых пунктов;

2. собирать и отображать ключевые данные;

3. позволять вручную загружать недостающую информацию;

4. импортировать данные из Excel/CSV;

5. учитывать поступившие заявки;

6. анализировать потенциальную абонентскую базу;

7. учитывать техническую возможность подключения;

8. считать стоимость строительства;

9. рассчитывать окупаемость;

10. анализировать конкурентов;

11. оценивать риски;

12. показывать достоверность данных;

13. сравнивать населённые пункты между собой;

14. формировать итоговую рекомендацию.

Итоговая рекомендация может быть одной из следующих:

- подключать;

- подключать в первую очередь;

- подключать позже;

- перспективно, но нужны уточнения;

- требуется техническое обследование;

- подключение возможно только при снижении затрат или росте заявок;

- подключение нецелесообразно.


## 2. Главная цель приложения

Главная цель приложения — помочь сотруднику интернет-провайдера быстро и обоснованно понять:

1. есть ли спрос в населённом пункте;

2. сколько потенциальных абонентов можно подключить;

3. есть ли техническая возможность строительства сети;

4. сколько потребуется инвестиций;

5. за сколько месяцев проект окупится;

6. есть ли конкуренты;

7. какие риски есть у проекта;

8. насколько достоверны исходные данные;

9. какой населённый пункт выгоднее подключать первым.

Приложение не должно быть просто базой данных.  
Оно должно быть инструментом для принятия управленческого решения.


## 3. Основные пользователи приложения

### 3.1. Менеджер по развитию сети

Использует приложение для первичной оценки населённого пункта.

Задачи:

- добавить населённый пункт;

- внести общие данные;

- загрузить заявки;

- оценить спрос;

- посмотреть предварительную рекомендацию;

- подготовить данные для руководителя.


### 3.2. Технический специалист

Заполняет техническую часть.

Задачи:

- указать расстояние до ближайшей сети;

- оценить наличие опор;

- определить сложность строительства;

- рассчитать предварительную длину линии;

- внести стоимость монтажных работ;

- указать техническое решение;

- оставить инженерный комментарий.


### 3.3. Руководитель

Смотрит итоговый отчёт и принимает решение.

Задачи:

- сравнить населённые пункты;

- посмотреть рейтинг;

- проверить окупаемость;

- оценить риски;

- увидеть достоверность данных;

- принять решение о запуске проекта.


### 3.4. Отдел продаж

Добавляет данные по спросу.

Задачи:

- загружать заявки;

- вносить обращения жителей;

- отмечать подтверждённые заявки;

- фиксировать результаты обзвона;

- указывать готовность жителей оплатить подключение;

- отмечать инициативные группы.


## 4. Первая версия приложения — MVP

Первая версия должна быть простой, быстрой в разработке и удобной для тестирования.

В первой версии **не делаем карту**, чтобы приложение было легче, проще и быстрее.

### В MVP входит:

1. список населённых пунктов;

2. карточка населённого пункта;

3. ручной ввод данных;

4. импорт заявок из Excel/CSV;

5. импорт технических данных;

6. импорт данных по конкурентам;

7. расчёт инвестиций;

8. расчёт выручки;

9. расчёт прибыли;

10. расчёт окупаемости;

11. расчёт стоимости подключения одного абонента;

12. сценарный расчёт;

13. итоговый рейтинг;

14. оценка достоверности данных;

15. приоритизация населённых пунктов;

16. текстовая рекомендация;

17. экспорт отчёта;

### Во вторую версию переносим:

1. карту;

## 14.1. Формулы

### Ежемесячная выручка

Ежемесячная выручка = количество абонентов × средний тариф

### Чистый доход

Чистый доход = ежемесячная выручка - ежемесячные расходы

### Окупаемость

Окупаемость = общие инвестиции / чистый ежемесячный доход

Если чистый ежемесячный доход меньше или равен нулю, приложение должно показывать:

**Проект не окупается при заданных параметрах.**

### Стоимость подключения одного абонента

Стоимость подключения одного абонента = общие инвестиции / прогнозируемое количество абонентов

### Минимальное количество абонентов для окупаемости

Минимальное количество абонентов = общие инвестиции / желаемый срок окупаемости / средний доход с абонента


# 15. Блок 8. Сценарный финансовый расчёт

В приложении нужно добавить три сценария расчёта:

1. пессимистичный;

2. базовый;

3. оптимистичный.

Это позволит руководителю видеть не один расчёт, а диапазон возможных результатов.


## 15.1. Пессимистичный сценарий

Используется, если подключится меньше абонентов, расходы будут выше, а сроки окупаемости длиннее.

Пример условий:

- подключится 15–20% домов;

- стоимость строительства увеличится на 10–20%;

- ежемесячные расходы будут выше плановых;

- часть заявок не подтвердится;

- конкуренты начнут удерживать клиентов скидками.


## 15.2. Базовый сценарий

Используется как основной реалистичный прогноз.

Пример условий:

- подключится 25–35% домов;

- стоимость строительства соответствует предварительной смете;

- средний тариф соответствует текущим тарифам провайдера;

- часть заявок перейдёт в реальные подключения.


## 15.3. Оптимистичный сценарий

Используется, если спрос высокий, конкуренты слабые, а техническая реализация проще ожидаемой.

Пример условий:

- подключится 40–50% домов;

- стоимость строительства не превысит смету;

- жители готовы подключаться активно;

- есть инициативная группа;

- есть поддержка администрации.


## 15.4. Поля сценарного расчёта

Для каждого сценария приложение должно считать:

- процент подключения домов;

- количество абонентов;

- средний тариф;

- ежемесячную выручку;

- ежемесячные расходы;

- чистый доход;

- инвестиции;

- срок окупаемости;

- стоимость подключения одного абонента.


# 16. Блок 9. Риски

Риски оцениваются по шкале от 1 до 5.

1 — низкий риск.  
5 — высокий риск.

### Примеры рисков:

- мало заявок;

- высокая стоимость строительства;

- сильные конкуренты;

- отсутствие разрешений;

- сложная трасса;

- дорогое обслуживание;

- низкая платёжеспособность;

- сезонное проживание;

- удалённость от действующей сети;

- отсутствие точных данных;

- высокая стоимость аренды опор;

- низкая плотность домов;

- риск неподтверждения заявок.

### Поля риска:

- тип риска;

- уровень риска;

- комментарий;

- кто внёс оценку;

- дата оценки.

Итоговая оценка рисков должна уменьшать общий рейтинг.


# 17. Блок 10. Оценка достоверности данных

Приложение должно показывать не только итоговый рейтинг, но и уровень доверия к расчёту.

Это важно, потому что нельзя принимать решение о строительстве сети только на основании неполных или приблизительных данных.


## 17.1. Уровни достоверности

### Высокая достоверность

Данные можно использовать для принятия предварительного управленческого решения.

Условия:

- заполнены заявки;

- есть подтверждённые адреса;

- есть техническая оценка;

- есть предварительная смета;

- есть данные по конкурентам;

- рассчитана экономика;

- заполнены риски.


### Средняя достоверность

Расчёт можно использовать для предварительного анализа, но перед запуском проекта нужны уточнения.

Условия:

- есть часть заявок;

- есть примерная стоимость;

- нет полного технического обследования;

- данные по конкурентам частичные;

- часть данных заполнена приблизительно.


### Низкая достоверность

Решение принимать нельзя, нужно собрать недостающую информацию.

Условия:

- мало подтверждённых заявок;

- нет технической оценки;

- нет сметы;

- нет данных по конкурентам;

- много полей заполнено приблизительно.


## 17.2. Пример вывода

Итоговый рейтинг: 76 из 100  
Рекомендация: перспективно, требуется уточнение данных  
Достоверность анализа: средняя

Причина: есть подтверждённый спрос и предварительная экономика, но отсутствует полное техническое обследование.


# 18. Логика итогового рейтинга

Приложение должно считать общий балл целесообразности подключения по шкале от 0 до 100.

### Вес критериев:

| **Критерий** | Вес |
| :-: | -: |
| Подтверждённый спрос | 25% |
| Потенциальная абонентская база | 15% |
| Техническая возможность | 20% |
| Стоимость строительства | 15% |
| Окупаемость | 15% |
| Конкуренция | 5% |
| Риски | 5% |


## 18.1. Оценка спроса

| **Условие** | Балл |
| :-: | -: |
| Нет заявок | 0 |
| 1–5 заявок | 20 |
| 6–20 заявок | 40 |
| 21–50 заявок | 70 |
| Более 50 заявок | 100 |

### Корректировки:

- если более 50% заявок подтверждены, добавить 10 баллов;

- если много дублей или неподтверждённых обращений, снизить на 10–20 баллов;

- если есть инициативная группа жителей, добавить 10 баллов;

- если есть готовность оплатить подключение, добавить 10 баллов.

Максимальный балл по спросу — 100.


## 18.2. Оценка потенциальной абонентской базы

| **Условие** | Балл |
| :-: | -: |
| Менее 30 домов | 20 |
| 30–100 домов | 40 |
| 101–300 домов | 70 |
| Более 300 домов | 100 |

### Корректировки:

- если большая часть домов сезонного проживания, снизить балл;

- если есть новые жилые массивы, повысить балл;

- если есть социальные и бизнес-объекты, повысить балл.


## 18.3. Оценка технической возможности

| **Условие** | Балл |
| :-: | -: |
| Техническая возможность отсутствует | 0 |

| Возможность есть, но нужны согласования | 60 |
| Возможность есть, сложность средняя | 80 |
| Подключение технически простое | 100 |


## 18.4. Оценка стоимости строительства

Основной показатель:

Стоимость подключения одного абонента = общие инвестиции / прогнозируемое количество абонентов

| **Условие** | Балл |

# 19. Приоритизация населённых пунктов

Приложение должно не только анализировать один населённый пункт, но и сравнивать несколько населённых пунктов между собой.

Для этого добавляется раздел:

**Очередность подключения**


## 19.1. Поля для приоритизации

- населённый пункт;

- район;

- итоговый рейтинг;

- количество подтверждённых заявок;

- потенциальная абонентская база;

- сумма инвестиций;

- срок окупаемости;

- техническая сложность;

- уровень конкуренции;

- уровень рисков;

- достоверность данных;

- рекомендуемая очередь подключения.


## 19.2. Варианты очередности

- 1 очередь — подключать в первую очередь;

- 2 очередь — перспективно, но нужно уточнить данные;

- 3 очередь — возможно подключение при росте спроса или снижении затрат;

- отложить — требуется дополнительное обследование;

- нецелесообразно — проект пока невыгоден.


## 19.3. Пример логики

Если у населённого пункта высокий спрос, подтверждённые заявки, короткая окупаемость и техническая возможность подключения, он попадает в первую очередь.

Если спрос есть, но не хватает технических данных или высокая стоимость строительства, населённый пункт попадает во вторую очередь.

Если заявок мало, конкуренты сильные, а строительство дорогое, населённый пункт откладывается.

# 22. Работа с интернет-данными

В первой версии интернет-поиск можно не делать обязательным.

На первом этапе лучше сделать ручной ввод и импорт таблиц.

Во второй версии можно добавить отдельный модуль интернет-поиска.


## 22.1. Какие данные можно искать автоматически во второй версии

| **Категория** | **Что искать** |
| :-: | :-: |
| Население | Численность жителей, административная принадлежность |
| География | Координаты, расстояние до ближайшего узла или города |
| Застройка | Частный сектор, плотность улиц, наличие СНТ/ИЖС |
| Конкуренты | Провайдеры, работающие в населённом пункте |
| Спрос | Жалобы, обсуждения, упоминания о плохом интернете |
| Социальные объекты | Школы, администрация, ФАП, магазины |
| Бизнес-объекты | Магазины, организации, предприятия |


## 22.2. Важное правило

Все найденные в интернете данные считаются предварительными и не должны участвовать в финальном решении без подтверждения пользователем.

Статусы данных:

- найдено автоматически;

- подтверждено сотрудником;

- отклонено;

- требует проверки;

- внесено вручную;

- импортировано из файла.


# 23. Учет закрытой внутренней системы провайдера

Если у провайдера есть внутренняя закрытая система, первую версию лучше делать так:

1. приложение работает локально на компьютере или сервере компании;

2. данные можно импортировать вручную через Excel/CSV;

3. внутренние данные не отправляются во внешние сервисы;

4. отчёты сохраняются локально;

5. доступ ограничен логином и паролем;

6. роли пользователей ограничивают права редактирования;

7. автоматический интернет-поиск делается отдельным модулем во второй версии.

# 30. Логика подсветки недостающих данных

Приложение должно показывать, каких данных не хватает.

Пример:

Для точного анализа не хватает:

- количества домовладений;

- подтверждённых заявок;

- расстояния до ближайшей сети;

- стоимости монтажа;

- данных по конкурентам;

- технического комментария;

- оценки рисков.

Это важно, потому что решение о подключении нельзя принимать только на основании неполных данных.


# 31. Пример итоговой рекомендации

Пример вывода:

**Рекомендация: подключение перспективно.**

Причины:

- собрано 47 заявок;

- 34 заявки подтверждены;

- первая очередь охватывает 180 домов;

- конкуренты представлены слабо;

- предварительная окупаемость — 14 месяцев;

- техническая возможность есть, но требуется согласование опор;

- достоверность данных — средняя.

Рекомендуемое действие:

Провести техническое обследование и начать сбор предварительных договоров по улицам Центральная, Школьная и Молодёжная.

# 34. Главный принцип приложения

Приложение должно помогать сотруднику провайдера ответить на главный вопрос:

**Стоит ли вкладывать деньги в подключение этого населённого пункта, в какой очередности это делать и при каких условиях проект станет выгодным?**

Первая версия должна быть не перегруженной, поэтому карта, автоматическая трассировка и сложная геоаналитика переносятся во вторую версию.
